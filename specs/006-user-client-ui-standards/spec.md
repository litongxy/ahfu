# Feature Specification: 用户端 UI 规范整理

**Feature Branch**: `006-user-client-ui-standards`  
**Created**: 2026-03-31  
**Status**: Draft  
**Input**: 用户端截图资料与现有推荐链路实现

## User Scenarios & Testing

### User Story 1 - 统一全量用户端视觉语言 (Priority: P1) 🎯 MVP

设计、研发、运营可以基于同一份规范理解用户端页面应该如何长什么样。

**Why this priority**: 先统一整体视觉语言，才能避免新页面和老页面风格割裂，提升品牌一致性。

**Independent Test**: 选取任意 5 个高频页面，对照规范检查页面结构、主色、标题层级、按钮样式和卡片样式是否一致。

**Acceptance Scenarios**:

1. **Given** 首页、商城、心理、睡眠等页面，**When** 对照规范检查，**Then** 这些页面共享统一的绿色品牌基调、标题层级和卡片语言
2. **Given** 同一业务的多个状态页，**When** 对照规范检查，**Then** 成功、失败、空状态和加载状态都遵循统一的版式原则
3. **Given** 不同模块的主按钮和次按钮，**When** 对照规范检查，**Then** 按钮语义、尺寸和视觉强调保持一致

---

### User Story 2 - 统一核心业务模块页面结构 (Priority: P1) 🎯 MVP

用户可以在首页、商城、精准健康、心理、睡眠、个人中心等主模块中，快速找到内容、操作和结果。

**Why this priority**: 页面结构统一后，用户学习成本更低，业务链路更清晰，后续页面也更容易扩展。

**Independent Test**: 对照规范检查每个核心模块是否定义了页面目标、核心区块、主要 CTA、跳转关系和常见状态。

**Acceptance Scenarios**:

1. **Given** 首页类页面，**When** 查看规范，**Then** 至少包含顶部信息区、推荐入口、模块快捷入口和底部导航规则
2. **Given** 商城类页面，**When** 查看规范，**Then** 至少包含搜索、分类、商品卡、筛选、确认订单和支付相关结构
3. **Given** 心理和睡眠类页面，**When** 查看规范，**Then** 至少包含内容封面、目录/介绍、课程播放、购买/参与 CTA 和结算态

---

### User Story 3 - 统一推荐链路呈现 (Priority: P2)

推荐内容、推荐商品和页面入口在不同场景下都能以统一方式展示。

**Why this priority**: 现有代码已经支持场景推荐，如果 UI 规范不统一，推荐链路会显得散乱且难以复用。

**Independent Test**: 选取饮食、运动、心理、睡眠、抗衰 5 个场景，对照规范检查推荐卡片、商品卡和页面入口是否有统一展示规则。

**Acceptance Scenarios**:

1. **Given** 饮食场景推荐，**When** 用户查看推荐区，**Then** 展示内容卡、商品卡和页面入口的统一组合
2. **Given** 运动、心理、睡眠场景推荐，**When** 用户查看推荐区，**Then** 不同场景仅更换内容，不改变推荐区的布局语言
3. **Given** 推荐内容不足，**When** 用户查看推荐区，**Then** 规范要求展示默认推荐或空状态引导

---

### User Story 4 - 统一通用组件与状态规范 (Priority: P2)

表单、弹窗、选择器、列表、空状态和权限提示在全站保持一致。

**Why this priority**: 用户端大量页面依赖这些通用组件，一旦统一，能直接提升整体体验和开发效率。

**Independent Test**: 任意抽查 10 个截图中的交互组件，核对它们是否满足统一的状态和样式规则。

**Acceptance Scenarios**:

1. **Given** 日期、城市、门店、规格、设备等选择场景，**When** 查看规范，**Then** 选择器组件有统一的交互层级和确认方式
2. **Given** 登录、支付、相册、蓝牙、相机等权限场景，**When** 查看规范，**Then** 权限弹窗有统一的文案结构和按钮语义
3. **Given** 列表为空或数据失败，**When** 查看规范，**Then** 空状态、错误态和加载态都有统一的视觉与引导策略

---

### Edge Cases

- 页面内容过长时，应保持顶部信息固定或分层折叠，避免首屏信息被压缩
- 商品、文章、课程、医生名称过长时，应定义截断和多行展示规则
- 订单、评价、问诊记录等长列表页面应定义分页/滚动/空状态规则
- 未登录、无权限、已下架、已失效、已完成等状态需要分别定义展示方式
- 同一页面在小屏与大屏机型上的安全区和底部操作区需要保持可用

## Requirements

### Functional Requirements

- **FR-001**: 系统 MUST 定义全量用户端页面的统一视觉语言，包括颜色、排版、间距、卡片和按钮风格
- **FR-002**: 系统 MUST 为首页、商城、精准健康、心理、睡眠、个人中心等核心模块定义页面级结构规则
- **FR-003**: 系统 MUST 为推荐内容、推荐商品和页面入口定义统一的展示规则
- **FR-004**: 系统 MUST 为表单、弹窗、选择器、列表、空状态和权限提示定义通用组件规范
- **FR-005**: 系统 MUST 覆盖用户端常见业务状态，包括空数据、加载中、成功、失败、下架、未登录、无权限和已完成
- **FR-006**: 系统 MUST 明确各页面的主要 CTA、次要 CTA 和跳转关系
- **FR-007**: 系统 MUST 明确长文、长标题、长列表和多层弹窗的展示与处理规则
- **FR-008**: 系统 MUST 让截图中的高频交互在规范中可被直接对照和复用
- **FR-009**: 系统 MUST 兼容当前绿色品牌风格，同时收束现有页面的视觉差异
- **FR-010**: 系统 MUST 为后续新增页面提供可复用的规范框架

### Key Entities

- **PagePattern**: 页面模式，描述页面目标、信息层级、核心区块、CTA 和跳转关系
- **ComponentPattern**: 通用组件模式，描述按钮、卡片、表单、弹窗、选择器和列表的展示规则
- **StatePattern**: 状态模式，描述空、加载、成功、失败、权限、下架、完成等状态
- **RecommendationSurface**: 推荐展示面，描述内容卡、商品卡和页面入口在推荐区中的组合方式

## Success Criteria

### Measurable Outcomes

- **SC-001**: 100% 的高频用户端页面可以在规范中找到对应页面模式
- **SC-002**: 至少 90% 的截图交互组件可以映射到统一组件规范
- **SC-003**: 至少 5 个推荐场景都拥有一致的推荐展示规则
- **SC-004**: 新增页面在评审时无需临时决定视觉语言，规范可直接作为统一依据
- **SC-005**: 设计和研发对同一页面结构的理解偏差显著减少，评审返工次数下降

## Assumptions

- 以用户端现有截图资料作为主要事实来源，当前绿色品牌风格作为基线
- 本次产出为页面级 UI 规范，不包含完整像素级设计稿或切图交付
- 不修改后端接口实现，只整理和统一用户端页面规范
- 规范优先覆盖高频页面和高频交互，其余页面按同类模式复用
